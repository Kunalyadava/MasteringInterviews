Angular Production Build Files

Understanding the files generated when running `ng build --prod`

Overview

When you build an Angular application using the command ng build --prod, Angular generates a set of files that are optimized for production. These files are necessary for the browser to run the application, and they differ from a development build in terms of optimization, size reduction, and caching improvements.

Files Generated in a Production Build

Here’s a breakdown of the most common files generated in the production build of an Angular application:

1. index.html

This is the main HTML file and the entry point for your Angular application. It includes references to the generated JavaScript and CSS files, which will be loaded when the application runs.

2. main.js

This is the primary JavaScript file that bootstraps the Angular application. It contains the Angular framework core, application modules, and other necessary code to initialize the app.

3. polyfills.js

This file contains polyfills that are necessary to ensure the application works across different browsers, particularly older browsers.

4. runtime.js

The runtime.js file is responsible for loading the other JavaScript bundles generated by Angular, and it ensures modules are loaded in the correct order.

5. vendor.js

This file contains third-party libraries and Angular’s core modules. It allows for better caching since this file changes less frequently than the application code.

6. styles.css (or styles.[hash].css)

This file contains the global styles for the application. In a production build, it’s often minified and hashed to enable better caching.

7. scripts.js (optional)

If you specified external JavaScript files, they will be bundled into this file.

8. [chunk-name].[hash].js (Lazy-loaded Modules)

These files represent lazy-loaded modules that are only loaded when needed, improving the performance of the application by reducing initial load time.

9. assets/ Folder

This folder contains static files such as images, fonts, and other media, which are copied from the source to the production build without being bundled into JavaScript.

10. favicon.ico

This is the small icon that appears in the browser tab. You can replace it with a custom favicon.

11. manifest.webmanifest (Optional, if PWA)

If your app is configured as a Progressive Web App (PWA), this file will contain metadata, including icons and theme colors, enabling users to install the app on their devices.

Example Directory Structure

Here is an example of what the production build might look like in the dist/ directory after running ng build --prod: 

dist/
├── index.html
├── main.[hash].js
├── polyfills.[hash].js
├── runtime.[hash].js
├── vendor.[hash].js
├── styles.[hash].css
├── [lazy-loaded-chunk].[hash].js
├── assets/
│   └── ...
└── favicon.ico

Summary

Each of the files generated by Angular in the production build has a specific role:

  • index.html serves as the entry point for the application.
  • JavaScript files (e.g., main.js, vendor.js) contain the code needed to run the app.
  • CSS files provide global styling for the app.
  • Assets include static resources like images and fonts.

In a production build, these files are minified, hashed, and optimized for faster load times and better caching, ensuring that your Angular application is both efficient and performant in production.
